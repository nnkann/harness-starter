# /advisor 스킬

판단이 어려울 때, 3개 에이전트가 병렬로 검증하고 근거를 제공한다.

## 사용법

| 사용법 | 설명 |
|--------|------|
| `/advisor <질문>` | 질문에 대해 3관점 검증 |
| `/advisor <계획 또는 접근법>` | 계획의 타당성 검증 |

## 핵심 원칙

- advisor는 **조언만** 한다. 코드를 수정하지 않는다.
- 결정은 **사용자가** 내린다.
- 에이전트는 **병렬로** 실행된다 (독립 관점 보장).
- 에이전트 1개가 실패해도 나머지 결과로 진행한다.

## 흐름

### Step 1. 질문/계획 수신

사용자의 질문 또는 계획을 받는다.
모호하면 한 줄로 "무엇을 검증할까요?"라고 물어본다.

### Step 2. 3개 에이전트 병렬 호출

**반드시 단일 메시지에서 3개 Agent 호출을 동시에** 보낸다.
순차 호출하면 앞 결과가 뒤를 오염시킨다.

#### 에이전트 1: 리서치 (Research) — 관점: 옹호자

> 역할: 외부 근거로 접근법을 **설득**한다.

이 에이전트는 제안된 접근법이 왜 합리적인지를 외부 자료로 뒷받침하는 옹호자 역할이다.
단, 근거 없는 긍정은 금지. 출처가 없으면 "근거를 찾지 못했다"고 보고한다.

조사 순서:
1. **공식 문서 우선** — Context7 MCP(`query-docs`)로 관련 라이브러리/프레임워크의 **최신 공식 문서**를 먼저 조회한다. 모델의 학습 데이터는 오래될 수 있으므로, 공식 문서가 항상 1순위 출처다.
2. **웹서칭 보완** — WebSearch로 공식 문서에 없는 커뮤니티 사례, 논문, 비교 분석을 추가 조사한다.
3. **최신 상태 확인** — deprecated 여부, 최근 릴리스 날짜, 커뮤니티 활성도, 알려진 문제점.
4. **성공 사례** — 유사 규모/도메인에서 이 접근법을 채택한 사례와 그 결과.

**출력**: "이 접근법이 합리적인 이유" + 근거 + 출처(공식 문서 링크, 웹 링크).

#### 에이전트 2: 코드 분석 (Codebase) — 관점: 개선자

> 역할: 현재 코드베이스 관점에서 **더 나은 방법**을 찾는다.

이 에이전트는 단순 정합성 검증이 아니라, 프로젝트의 기존 코드·패턴·구조·문서를 기반으로
"이미 있는 것을 활용하면 더 낫다", "기존 패턴을 따르면 이렇게 된다"는 개선 제안을 한다.

탐색 순서:
1. **docs/ 클러스터 탐색 (먼저)** — `docs/INDEX.md` → 관련 도메인의 `clusters/{domain}.md` → 본문 Read. 특히:
   - `decisions/`: 과거에 같은 영역에서 내린 결정과 그 근거 (선례)
   - `guides/`: 기존 패턴과 방법론 (따라야 할 것)
   - `relates-to` 포인터로 연관 문서 1홉 추가 탐색
2. **코드베이스 탐색** — Explore 서브에이전트로 실제 코드 확인
   - 기존 패턴, 아키텍처, 컨벤션과의 충돌 여부
   - 유사 기능이 이미 존재하는지 확인 (재사용 기회)
   - 영향 범위 분석 (이 변경이 어디까지 퍼지는가)
3. **문서 ↔ 코드 대조** — decisions/의 근거가 현재 코드에서도 유효한지, 제안이 기존 결정과 충돌하지 않는지 확인

**출력**: "현재 코드베이스에서 이렇게 하면 더 낫다" + 코드 위치·문서 위치를 가리키는 구체적 근거.

#### 에이전트 3: 리스크 (Risk) — 관점: 비판자

> 역할: 무조건 **반대 논거**를 찾는다. Devil's advocate.

이 에이전트는 접근법의 약점을 적극적으로 찾는 비판자다.
"괜찮아 보인다"는 결론이 나오더라도, 억지로라도 반론을 만들어야 한다.
모든 접근법에는 트레이드오프가 있고, 이 에이전트의 역할은 그것을 드러내는 것이다.

탐색 순서:
1. **docs/ 과거 사례 확인 (먼저)** — `docs/INDEX.md` → 관련 도메인의 `clusters/{domain}.md` → 본문 Read. 특히:
   - `incidents/`: 과거에 비슷한 접근법으로 발생한 장애·버그 (반복 방지)
   - `decisions/`: 과거 결정에서 기록된 트레이드오프와 "선택하지 않은 대안" (실패 근거)
2. **리스크 분석** — 과거 사례 + 코드 분석을 결합하여:
   - 보안 취약점 (OWASP Top 10 등)
   - 성능 병목 가능성
   - 유지보수 비용 (복잡도, 의존성, 기술 부채)
   - 롤백 가능성 (되돌리기 어려운 결정인가)
   - 6개월 뒤 이 결정을 후회할 시나리오
   - 이 접근법을 채택하지 **않는** 이유를 최소 1개 제시
   - **과거 incidents/에 유사 패턴이 있으면 반드시 언급**

**출력**: "이 접근법의 문제점" + 리스크 목록 + 심각도 + 완화 방안 + 관련 과거 사례(있으면).

### Step 3. 결과 종합

3개 결과를 종합하여 아래 형식으로 보고한다.

```
## /advisor 결과

### 추천
[3개 분석을 종합한 추천 사항. 1~3줄.]

### 옹호 (리서치)
[외부 근거로 이 접근법이 합리적인 이유. 공식 문서 + 웹 링크 포함.]

### 개선 (코드 분석)
[현재 코드베이스에서 더 나은 방법. 파일 위치 포함.]

### 비판 (리스크)
[이 접근법의 문제점. 심각도 포함.]

### 대안
[추천과 다른 접근법. 있으면.]
```

3개 관점이 모두 일치하면: "3관점 일치. 진행 추천. ✅"
관점 간 충돌이 있으면: 충돌을 명시하고 사용자에게 선택을 요청한다.

## 언제 사용하는가

- 기술 스택 선택이 고민될 때
- 아키텍처 결정에 확신이 없을 때
- 새로운 라이브러리 도입을 고려할 때
- 리팩토링 방향이 여러 개일 때
- "이게 맞는 건가?" 싶을 때

## 언제 사용하지 않는가

- 답이 명확한 단순 질문 (advisor 호출 = 추가 토큰 비용)
- 이미 결정된 사항의 재확인
- 코드 스타일/네이밍 같은 컨벤션 문제 (naming.md, coding.md 참조)

## 다른 스킬과의 연동

| 스킬 | 연동 지점 | 동작 |
|------|----------|------|
| implementation | Step 0.5 | "접근법 검증할까요? [Y/n]" → Y 시 advisor 호출 |
| harness-init | Step 6 | 스택 결정 전 리서치 에이전트로 최신 공식 문서 + 상태 확인 |
| eval --deep | 2차 검증 | 3관점을 병렬 에이전트로 실행 |
| commit --strict | Review | 리뷰를 독립 에이전트로 분리 |
