# 하네스 초기화

PRD 또는 아이디어를 입력받아, 대화로 기술 결정을 하나씩 좁혀간다.
모든 결정은 사용자가 내린다. Claude는 선택지와 근거를 제시할 뿐이다.

## 전제

- 신규 프로젝트 전용. 기존 프로젝트 적용은 harness-adopt 스킬(별도)을 사용한다.
- 하네스 파일 구조(.claude/rules/, skills/, scripts/)는 이미 존재한다.
- 이 스킬은 **빈 칸을 채우는 과정**이다. 구조를 새로 만들지 않는다.

## 흐름

### Step 1. 요구사항 분석

사용자에게 PRD, 기획 문서, 또는 구두 설명을 요청한다.
받으면 다음 순서로 추출하고 요약 확인한다.

**1) 한 줄 정의** — 이 제품/서비스는 무엇인가.

**2) 페르소나 식별** — 누가 쓰는가. 유형별로 분리한다.
- 각 페르소나가 이 제품에서 하는 일이 다르면 별개 페르소나.
- 페르소나가 1개뿐이면 그대로 진행. 억지로 나누지 않는다.

**3) 페르소나별 핵심 기능** — 각 페르소나가 필요한 기능을 나열한다.
- 페르소나와 연결되지 않는 기능은 의심한다.

**4) MECE 검증** — 빠진 것과 겹친 것을 확인한다.
- 완벽할 필요 없다. 명백한 누락과 중복만 잡는다.

추출이 모호하면 질문한다. 추측하지 않는다.

### Step 2. CPS 정리

기술 결정 전에 "왜 만드는가"를 구조화한다.

> **CPS 행동 원칙 (이 단계에서 적용)**
> - 사용자가 말한 것을 정리하는 것이 아니라, **안 말한 것을 찾아내는 것**이 핵심이다.
> - 질문을 던져라: "이 페르소나가 이 상황에서 이것도 필요하지 않나?"
> - 빈 곳을 찾아라: Problem 없는 Solution, Solution 없는 Problem을 지적하라.
> - 도전하라: "이 기능이 정말 MVP에 필요한가?" "이 제약은 진짜 제약인가, 가정인가?"
> - 동의하는 척하지 마라. 근거 있으면 반박하라.
> - 한 번에 완성하려 하지 마라. 여러 번 오갈 수 있다.

**Context** — 배경. 이 프로젝트가 존재하는 이유.
- Step 1의 페르소나들이 어떤 상황에서 이것을 필요로 하는가?
- 기존에 어떻게 해결하고 있는가?
- 어떤 제약이 있는가? (시간, 인력, 예산, 기술)
- **프로젝트 중요도(stakes)** — 하네스 강도를 결정한다.

| 강도 | 기준 | 예시 | 적용 |
|------|------|------|------|
| light | 일회성, 프로토타입, 학습용 | 해커톤, 개인 실험, PoC | Phase 1만. 리뷰 최소. |
| standard | 사용자가 있지만 소규모 | 사이드 프로젝트, MVP | Phase 1~2. 핵심 경로 리뷰. |
| strict | 장기 유지보수, 사용자 다수 | SaaS, 오픈소스, 엔터프라이즈 | Phase 1~3 전부. 전체 리뷰. |

**Problem** — 구체적 문제.
- 해결해야 할 핵심 문제를 1~3개로 정의한다.
- "~하기 어렵다", "~할 수 없다" 형태로 명확하게.
- 문제 간 우선순위를 매긴다.

**Solution** — 대응 방안.
- 각 Problem에 대해 제품이 제공하는 해결책을 정리한다.
- Step 1의 페르소나별 핵심 기능과 연결되어야 한다.
- 연결이 안 되는 기능이 있으면 → 불필요한 기능이거나 Problem이 빠진 것.

> **강제력 설계 (CPS에서 코드로 가는 다리)**
> Solution은 결국 코드가 된다. 코드는 하네스가 통제한다.
> Solution을 설계할 때 "이것을 어떻게 강제할 것인가"를 함께 생각하라.
> - Linter (물리 법칙) — 우회 불가. 린터로 잡을 수 있으면 린터에 넣어라.
> - Hooks (법) — 우회 불가. 커밋 전 검증, 세션 규칙 주입.
> - Rules (자동 로드) — 무시 가능하지만 항상 보임.
> - CLAUDE.md (힌트) — 가장 약함. 소원만 적어두는 건 Solution이 아니다.

CPS가 정리되면 사용자에게 확인받는다.
이 문서는 implementation 스킬에 따라 docs/wip/에 기록한다.

### Step 3. 프로젝트 유형

Step 1~2의 요약을 바탕으로 유형을 판단한다.

| 유형 | 예시 |
|------|------|
| 웹앱 (풀스택) | 대시보드, SaaS, 이커머스 |
| 웹앱 (프론트만) | 랜딩, 포트폴리오, 정적 사이트 |
| API 서버 | REST/GraphQL 백엔드, 마이크로서비스 |
| CLI 도구 | 커맨드라인 유틸리티 |
| 라이브러리/패키지 | npm/pip에 배포하는 모듈 |
| 모바일 | React Native, Flutter 등 |

### Step 4. 규모 판단

| 규모 | 기준 | 아키텍처 방향 |
|------|------|--------------|
| 소 | 도메인 1~2개, 단일 배포 | flat |
| 중 | 도메인 3~5개, 단일 배포 | feature-based |
| 대 | 도메인 5개+, 또는 다중 배포 | layered 또는 monorepo |

### Step 5. 아키텍처 패턴

Step 3 + 4 결과로 선택지를 좁힌다.
flat / feature-based / layered / monorepo.
추천 근거를 설명하고, 사용자가 선택한다.

### Step 6. 스택 결정

아키텍처가 정해지면 순서대로 결정한다.
**각 항목에서 2~3개 선택지 + 한 줄 근거를 제시한다.**

> **Standards 행동 원칙 (이 단계에서 적용)**
> - "요즘 다 이렇게 해요"는 근거가 아니다. 왜 표준이 됐는지, 트레이드오프를 함께.
> - 대기업 표준을 소규모에 강요하지 마라. "오버엔지니어링 아닌가?"를 항상 자문.
> - 유행과 표준을 구분하라. 6개월짜리 트렌드와 5년짜리 표준은 다르다.
> - 사용자가 "모르겠다"면 가장 무난한 것을 추천하되, 나중에 바꿀 수 있다고 말하라.
> - 확실하지 않으면 모른다고 하라. 오래된 정보를 최신처럼 말하지 마라.

#### 6a. 코어 스택

1. **언어/런타임** — 프로젝트 유형이 가장 강하게 좁힌다
2. **프레임워크** — 언어가 좁힌다
3. **패키지 매니저** — 언어/프레임워크가 좁힌다
4. **테스트 프레임워크** — 언어가 좁힌다
5. **린터/포매터** — 언어가 좁힌다
6. **배포 환경** — 프로젝트 유형 + 사용자 상황이 좁힌다

#### 6b. UI/디자인 스택

프로젝트 유형에 따라 깊이가 다르다.

1. **스타일링 접근법** — 유틸리티 vs CSS-in-JS vs 모듈 CSS vs 바닐라
2. **컴포넌트 기반** — 헤드리스 / 스타일드 / 조합형
3. **아이콘** — 프레임워크 생태계 호환성 확인.
4. **폰트** — 시스템 폰트 vs 웹폰트. 한글 지원 확인.
5. **애니메이션** — 없음 / CSS 전환만 / 라이브러리

> **디자인 패턴과 아키텍처 패턴 (Standards)**
> - 프로젝트 유형과 규모에 맞는 디자인 패턴을 추천하라.
> - 패턴 선택이 naming.md, 폴더 구조, 코딩 컨벤션에 영향을 주면 업데이트 제안.
> - 패턴은 도구다. 목적 없이 적용하지 마라.
> - 직접 구현 vs 라이브러리: 유지보수, 커뮤니티, 번들 크기, 대안을 함께.

### Step 7. 결정 기록 및 하네스 반영

모든 결정이 끝나면 두 가지를 한다.

**1) 결정 기록** — implementation 스킬에 따라 docs/wip/에 기록.

**2) 하네스 파일 반영**

| 결정 | 반영 대상 |
|------|----------|
| CPS (Context, Problem, Solution) | docs/wip/ 결정 문서 |
| 패키지 매니저, 빌드/실행 명령, 배포 방식 | CLAUDE.md `## 환경` |
| 아키텍처 패턴, 폴더 규칙 | naming.md `## 폴더명` |
| PRD에서 추출한 도메인 목록 | naming.md `## 도메인 목록` |
| 린터/포매터 | 프로젝트 루트 설정 파일 |
| 테스트 프레임워크 | CLAUDE.md 또는 .claude/rules/ |
| UI/디자인 스택 | CLAUDE.md `## 환경` |

**반영 전 사용자에게 요약을 보여주고 확인받는다.**

## 주의

- 이 스킬은 범용이다. 특정 언어나 프레임워크를 강제하지 않는다.
- Step 3~6에서 제시하는 선택지는 예시다. 사용자의 상황에 맞게 조정한다.
- "모르겠다"는 유효한 답이다.
