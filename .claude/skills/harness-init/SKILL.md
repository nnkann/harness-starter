# 하네스 초기화

PRD 또는 아이디어를 입력받아, 대화로 기술 결정을 하나씩 좁혀간다.
모든 결정은 사용자가 내린다. Claude는 선택지와 근거를 제시할 뿐이다.

## 전제

- 신규 프로젝트 전용. 기존 프로젝트 적용은 harness-adopt 스킬(별도)을 사용한다.
- 하네스 파일 구조(.claude/rules/, skills/, scripts/)는 이미 존재한다.
- 이 스킬은 **빈 칸을 채우는 과정**이다. 구조를 새로 만들지 않는다.

## Step 0. 기존 프로젝트 감지 (차단 게이트)

init 시작 전 **반드시** 다음을 확인한다:

1. 프로젝트 루트에 소스 코드 파일이 이미 존재하는가? (`src/`, `app/`, `lib/`, `*.py`, `*.ts`, `*.js`, `*.go` 등)
2. `docs/`에 하네스 구조가 아닌 기존 문서가 있는가?
3. `.claude/settings.json`에 하네스 것이 아닌 기존 hooks/permissions가 있는가?

**하나라도 해당하면** adopt 완료 여부를 확인한다:
- `.claude/HARNESS.json`에 `adopted_at` 필드가 존재하면 → adopt 완료. **게이트 통과, init 진행.**
- `adopted_at`이 없으면 → init을 실행하지 않고 중단:

> ⛔ 기존 프로젝트가 감지되었습니다.
>
> 감지된 항목:
>   - [감지된 항목 나열]
>
> 기존 프로젝트에는 `/harness-adopt`를 먼저 실행해야 합니다.
> adopt가 .claude/ 병합과 docs/ 재분류를 처리한 뒤, init으로 CPS와 스택을 채울 수 있습니다.
>
> 순서: `/harness-adopt` → `/harness-init`

**예외:** 사용자가 명시적으로 "이건 신규 프로젝트야"라고 확인한 경우에만 진행을 허용한다.
단순히 "진행해"라는 요청으로는 게이트를 통과하지 않는다. 기존 파일이 있는데 신규라면 그 이유를 설명해야 한다.

## 흐름

### Step 1. 요구사항 분석

사용자에게 PRD, 기획 문서, 또는 구두 설명을 요청한다.
받으면 다음 순서로 추출하고 요약 확인한다.

**1) 한 줄 정의** — 이 제품/서비스는 무엇인가.

**2) 페르소나 식별** — 누가 쓰는가. 유형별로 분리한다.
- 각 페르소나가 이 제품에서 하는 일이 다르면 별개 페르소나.
- 페르소나가 1개뿐이면 그대로 진행. 억지로 나누지 않는다.

**3) 페르소나별 핵심 기능** — 각 페르소나가 필요한 기능을 나열한다.
- 페르소나와 연결되지 않는 기능은 의심한다.

**4) MECE 검증** — 빠진 것과 겹친 것을 확인한다.
- 완벽할 필요 없다. 명백한 누락과 중복만 잡는다.

추출이 모호하면 질문한다. 추측하지 않는다.

### Step 2. CPS 정리

기술 결정 전에 "왜 만드는가"를 구조화한다.

> **CPS 행동 원칙 (이 단계에서 적용)**
> - 사용자가 말한 것을 정리하는 것이 아니라, **안 말한 것을 찾아내는 것**이 핵심이다.
> - 질문을 던져라: "이 페르소나가 이 상황에서 이것도 필요하지 않나?"
> - 빈 곳을 찾아라: Problem 없는 Solution, Solution 없는 Problem을 지적하라.
> - 도전하라: "이 기능이 정말 MVP에 필요한가?" "이 제약은 진짜 제약인가, 가정인가?"
> - 동의하는 척하지 마라. 근거 있으면 반박하라.
> - 한 번에 완성하려 하지 마라. 여러 번 오갈 수 있다.

**Context** — 배경. 이 프로젝트가 존재하는 이유.
- Step 1의 페르소나들이 어떤 상황에서 이것을 필요로 하는가?
- 기존에 어떻게 해결하고 있는가?
- 어떤 제약이 있는가? (시간, 인력, 예산, 기술)
- **프로젝트 중요도(stakes)** — 하네스 강도를 결정한다.

| 강도 | 기준 | 예시 | 적용 |
|------|------|------|------|
| light | 일회성, 프로토타입, 소규모 | 해커톤, 개인 실험, PoC, MVP | commit light. 리뷰 최소. |
| strict | 장기 유지보수, 사용자 다수 | SaaS, 오픈소스, 엔터프라이즈 | commit strict. 전체 리뷰. |

**기본값 없음.** 사용자가 반드시 둘 중 하나를 선택해야 한다. "모르겠다"면 `strict`를 추천하되(안전 쪽), 강제하지 않는다. 선택 결과는 Step 7에서 `CLAUDE.md` `## 환경`의 `하네스 강도:`에 기록한다.

**Problem** — 구체적 문제.
- 해결해야 할 핵심 문제를 1~3개로 정의한다.
- "~하기 어렵다", "~할 수 없다" 형태로 명확하게.
- 문제 간 우선순위를 매긴다.

**Solution** — 대응 방안.
- 각 Problem에 대해 제품이 제공하는 해결책을 정리한다.
- Step 1의 페르소나별 핵심 기능과 연결되어야 한다.
- 연결이 안 되는 기능이 있으면 → 불필요한 기능이거나 Problem이 빠진 것.

> **강제력 설계 (CPS에서 코드로 가는 다리)**
> Solution은 결국 코드가 된다. 코드는 하네스가 통제한다.
> Solution을 설계할 때 "이것을 어떻게 강제할 것인가"를 함께 생각하라.
> - Linter (물리 법칙) — 우회 불가. 린터로 잡을 수 있으면 린터에 넣어라.
> - Hooks (법) — 우회 불가. 커밋 전 검증, 세션 규칙 주입.
> - Rules (자동 로드) — 무시 가능하지만 항상 보임.
> - CLAUDE.md (힌트) — 가장 약함. 소원만 적어두는 건 Solution이 아니다.

CPS가 정리되면 사용자에게 확인받는다.
이 문서는 implementation 스킬에 따라 docs/WIP/에 기록한다.

### Step 3. 프로젝트 유형

Step 1~2의 요약을 바탕으로 유형을 판단한다.

| 유형 | 예시 |
|------|------|
| 웹앱 (풀스택) | 대시보드, SaaS, 이커머스 |
| 웹앱 (프론트만) | 랜딩, 포트폴리오, 정적 사이트 |
| API 서버 | REST/GraphQL 백엔드, 마이크로서비스 |
| CLI 도구 | 커맨드라인 유틸리티 |
| 라이브러리/패키지 | npm/pip에 배포하는 모듈 |
| 모바일 | React Native, Flutter 등 |

### Step 4. 규모 판단

| 규모 | 기준 | 아키텍처 방향 |
|------|------|--------------|
| 소 | 도메인 1~2개, 단일 배포 | flat |
| 중 | 도메인 3~5개, 단일 배포 | feature-based |
| 대 | 도메인 5개+, 또는 다중 배포 | layered 또는 monorepo |

### Step 5. 아키텍처 패턴

Step 3 + 4 결과로 선택지를 좁힌다.
flat / feature-based / layered / monorepo.
추천 근거를 설명하고, 사용자가 선택한다.

### Step 6. 스택 결정

아키텍처가 정해지면 순서대로 결정한다.
**각 항목에서 2~3개 선택지 + 한 줄 근거를 제시한다.**

> **Standards 행동 원칙 (이 단계에서 적용)**
> - "요즘 다 이렇게 해요"는 근거가 아니다. 왜 표준이 됐는지, 트레이드오프를 함께.
> - 대기업 표준을 소규모에 강요하지 마라. "오버엔지니어링 아닌가?"를 항상 자문.
> - 유행과 표준을 구분하라. 6개월짜리 트렌드와 5년짜리 표준은 다르다.
> - 사용자가 "모르겠다"면 가장 무난한 것을 추천하되, 나중에 바꿀 수 있다고 말하라.
> - 확실하지 않으면 모른다고 하라. 오래된 정보를 최신처럼 말하지 마라.

#### 스택 검증 에이전트 (선택지 제시 전)

선택지를 사용자에게 보여주기 전에 **리서치 에이전트를 호출**하여 후보 기술의 최신 상태를 확인한다.

- Context7 MCP(`query-docs`)로 후보 기술의 **최신 공식 문서**를 조회한다.
- WebSearch로 deprecated 여부, 최근 릴리스 날짜, 커뮤니티 활성도를 확인한다.
- "이 조합을 실제로 쓰고 있는 프로젝트" 사례를 검색한다.

이 검증은 Standards 원칙의 **"오래된 정보를 최신처럼 말하지 마라"를 실제로 강제**한다.
모델의 학습 데이터 컷오프 한계를 웹서칭으로 보완하는 것이 목적이다.

검증 결과에 따라 선택지를 조정한다:
- deprecated된 기술 → 선택지에서 제외하거나 경고 표시
- 최근 메이저 변경이 있는 기술 → 변경 내용 함께 안내

#### 6a. 코어 스택

1. **언어/런타임** — 프로젝트 유형이 가장 강하게 좁힌다
2. **프레임워크** — 언어가 좁힌다
3. **패키지 매니저** — 언어/프레임워크가 좁힌다
4. **테스트 프레임워크** — 언어가 좁힌다
5. **린터/포매터** — 언어가 좁힌다
6. **배포 환경** — 프로젝트 유형 + 사용자 상황이 좁힌다

#### 6b. UI/디자인 스택

프로젝트 유형에 따라 깊이가 다르다.

1. **스타일링 접근법** — 유틸리티 vs CSS-in-JS vs 모듈 CSS vs 바닐라
2. **컴포넌트 기반** — 헤드리스 / 스타일드 / 조합형
3. **아이콘** — 프레임워크 생태계 호환성 확인.
4. **폰트** — 시스템 폰트 vs 웹폰트. 한글 지원 확인.
5. **애니메이션** — 없음 / CSS 전환만 / 라이브러리

> **디자인 패턴과 아키텍처 패턴 (Standards)**
> - 프로젝트 유형과 규모에 맞는 디자인 패턴을 추천하라.
> - 패턴 선택이 naming.md, 폴더 구조, 코딩 컨벤션에 영향을 주면 업데이트 제안.
> - 패턴은 도구다. 목적 없이 적용하지 마라.
> - 직접 구현 vs 라이브러리: 유지보수, 커뮤니티, 번들 크기, 대안을 함께.

### Step 7. 결정 기록 및 하네스 반영

모든 결정이 끝나면 두 가지를 한다.

**1) 프로젝트 출범 문서 생성** — `docs/WIP/project_kickoff_{YYMMDD}.md`

이 문서는 **프로젝트의 시작점**이다. harness-init 대화에서 결정된 모든 것의 단일 기록지이자, 이후 모든 작업 문서의 발원지. 안내성 플레이스홀더가 아니라 **실제 결정으로 채워진 문서**여야 한다.

```markdown
---
title: 프로젝트 출범: [프로젝트명]
domain: meta
tags: [cps, stack]
relates-to: []
status: in-progress
created: {YYYY-MM-DD}
---

# 프로젝트 출범: [프로젝트명]

## 한 줄 정의
[Step 1 결과]

## 페르소나와 핵심 기능
[Step 1 결과 요약]

## CPS
### Context
[Step 2 — 배경, 제약, 프로젝트 중요도]

### Problem
[Step 2 — 핵심 문제 1~3개]

### Solution
[Step 2 — 대응 방안, 강제력 설계 포함]

## 기술 결정
- **프로젝트 유형**: [Step 3]
- **규모**: [Step 4]
- **아키텍처 패턴**: [Step 5]
- **하네스 강도**: [light 또는 strict — 사용자가 선택한 값]
- **언어/런타임**: [Step 6a]
- **프레임워크**: [Step 6a]
- **패키지 매니저**: [Step 6a]
- **테스트**: [Step 6a]
- **린터/포매터**: [Step 6a]
- **배포 환경**: [Step 6a]
- **UI/디자인 스택**: [Step 6b, 해당 시]

## 도메인 목록
[Step 1/5에서 도출된 도메인들]

## 구현 순서 (Phase 1)
[Step 8a에서 결정된 순서가 여기 채워진다]

## 메모
(프로젝트 진행 중 주요 결정 변화를 간단히 기록)
```

이 문서 생성 직후 **docs-manager 스킬을 호출**하여:
- `docs/clusters/{domain}.md` 초기 생성 (도메인별 문서 목록 — 진입점 SSOT)
- 첫 번째 엔트리 = project_kickoff 문서

호출 시 전달 규약 (누수 #5 해소):
- `trigger`: "harness-init Step 7 — CPS 문서 신규 생성 직후 첫 INDEX 구축"
- `intent`: `full-refresh` (INDEX·clusters 자체가 없으므로 신규 생성)
- `scope: full` (최초 1회 — 빈 docs/ 구조 생성)
- `files`: 방금 생성한 project_kickoff 경로 (action: created, domain·status 명시)
- `context.prior_steps`: "Step 1~6에서 스택·도메인 결정 완료, 본 Step 7에서
  CPS 문서 생성. 본 호출은 docs/ 초기 골격 구축이 목적"

docs-manager가 docs.md 규칙에 따라 구조를 생성하므로, init 스킬이 INDEX 형식을 직접 알 필요 없다.

이 문서는 프로젝트가 존재하는 한 유지된다. commit 스킬이 이동시키지 않는다. 커밋 시점에는 `status`를 `completed`가 아닌 `in-progress`(또는 장기 유지 시 `living`)로 유지하고, 해당 문서는 **수동으로 `docs/guides/`로 이동**시켜 프로젝트 루트 기록지로 보존한다.

**2) 하네스 파일 반영**

| 결정 | 반영 대상 |
|------|----------|
| CPS (Context, Problem, Solution) | docs/WIP/ 결정 문서 |
| 하네스 강도 (light/strict) | CLAUDE.md `## 환경` `하네스 강도:` |
| 패키지 매니저, 빌드/실행 명령, 배포 방식 | CLAUDE.md `## 환경` |
| 아키텍처 패턴, 폴더 규칙 | naming.md `## 폴더명` |
| PRD에서 추출한 도메인 목록 | naming.md `## 도메인 목록` |
| 린터/포매터 | 프로젝트 루트 설정 파일 |
| 테스트 프레임워크 | CLAUDE.md 또는 .claude/rules/ |
| UI/디자인 스택 | CLAUDE.md `## 환경` |
| 프로젝트 소개 | README.md (신규 생성) |

**3) README.md 생성**

CPS와 스택이 확정된 시점이므로 프로젝트 README를 생성한다.

```markdown
# [프로젝트명]

[Step 1의 한 줄 정의]

## 스택

- **언어/런타임**: [Step 6 결과]
- **프레임워크**: [Step 6 결과]
- **패키지 매니저**: [Step 6 결과]

## 시작하기

\```bash

**반영 전 사용자에게 요약을 보여주고 확인받는다.**

### Step 8. 첫 번째 작업 생성

초기화가 끝나면 `docs/WIP/`에 **첫 번째 작업 문서가 존재해야 한다.**
안내 메시지만 보여주고 끝내지 않는다. 문서를 만든다.

#### 8a. 구현 순서 확인

- Step 5의 도메인 목록에서 Phase 1 도메인을 나열한다.
- 의존 관계를 분석하고 구현 순서를 제안한다.
- 사용자에게 "어디서부터 시작할 건가?"를 확인받는다.

#### 8b. 첫 번째 작업 문서 생성

사용자가 선택한 도메인에 대해 `docs/WIP/`에 구현 계획 문서를 **직접 생성한다.**

파일명: `guides--{도메인명}_{YYMMDD}.md`

```markdown
---
title: [도메인명] 구현
domain: [도메인명]
tags: []
relates-to:
  - path: guides/project_kickoff_{YYMMDD}.md
    rel: implements
status: pending
created: {YYYY-MM-DD}
---

# [도메인명] 구현

## 목표
- [이 도메인에서 구현할 것. CPS Problem과 연결.]

## 범위
- [Phase 1에서 이 도메인이 담당하는 기능 목록.]
- [구체적으로. "인증 구현"이 아니라 "이메일/비밀번호 로그인, JWT 발급, 토큰 갱신".]

## 선행 조건
- [이 작업 전에 필요한 것. 프로젝트 스캐폴딩, DB 설정 등.]
- [없으면 "없음".]

## 결정 사항
(작업하면서 채움)

## 메모
(작업 중 발견한 것, 변경 이유 등)
```

이 문서가 존재해야 harness-init이 완료된 것이다.

#### 8c. 완료 처리

- Step 7의 결정 문서(CPS + 스택) status를 `completed`로 변경한다.
- 커밋 시 commit 스킬이 `docs/guides/`로 이동시킨다.
- `docs/guides/project_kickoff_sample.md`가 존재하면 삭제한다 (실제 CPS로 대체되었으므로).

#### 8d. 완료 메시지

```
✅ 하네스 초기화 완료.

📄 docs/WIP/:
  - guides--[도메인]_[날짜].md (pending) ← 첫 번째 작업

📋 다음: 이 문서의 status를 in-progress로 바꾸고 구현 시작.
   바로 시작하시겠습니까?
```

## 주의

- 이 스킬은 범용이다. 특정 언어나 프레임워크를 강제하지 않는다.
- Step 3~6에서 제시하는 선택지는 예시다. 사용자의 상황에 맞게 조정한다.
- "모르겠다"는 유효한 답이다.
