# No-Speculation 규칙

추측으로 수정을 시작하지 마라.

## 배경

CLAUDE.md에 "추측 금지"가 있어도 매번 "이것일 거"라고 예측하고 수정 진행.
sonnet 사용 시 이 확률이 크게 증가한다고 사용자가 명시적으로 지적.

텍스트 규칙만으로는 LLM 행동이 바뀌지 않는다. 자동 감지·차단 장치가
없으면 위반에 비용이 없어서 패턴이 반복된다. 이 규칙은 **review 에이전트가
감지**한다.

## 첫 행동 3원칙

문제에 대한 첫 행동은 다음 중 하나여야 한다:

1. **관찰** — 실제 상태를 직접 읽는다 (파일, 로그, git status, 실행 결과)
2. **재현** — 문제가 발생하는 최소 케이스를 만든다
3. **선행 사례 확인** — 내부 자료(`internal-first.md` 참조)에서 유사 사례

## 금지 행동

- "아마 X가 원인일 것"으로 시작해서 X를 바로 고치는 것
- 원문(에러 메시지·로그·소스)을 직접 보기 전에 해결책을 설계하는 것
- 테스트·실행 없이 "이제 될 것"이라고 결론내리는 것
- 사용자에게 "이걸로 될까요?"라고 검증 책임을 떠넘기는 것

## 확신 표현

확신이 없으면 **명시적으로** 표현한다:
- "가설이다, 확인 필요"
- "X일 가능성이 있는데, 로그를 봐야 확정"
- "지금까지 본 것만으로는 단정 불가"

확신 없는 추측을 단정형으로 쓰지 마라. 사용자가 그걸 사실로 믿고 다음
행동을 결정한다.

## 자동 감지 (review 에이전트)

review 에이전트가 다음 패턴을 검증:

1. **추측 단어 + 근거 없음**
   - 커밋 메시지·WIP·코드 주석에 "아마", "일 것 같은", "예상", "추정",
     "probably", "might be"가 등장하는데 그 옆에 근거(파일·라인·로그)가 없음
   - → 경고

2. **문제 설명 없는 수정**
   - diff가 "무엇이 깨졌는지" 설명 없이 바로 수정으로 구성
   - 커밋 메시지에 해결하려는 문제 기술이 없음
   - → 경고

3. **검증 없는 "완료" 선언**
   - 변경 후 빌드·테스트·실행 흔적 없이 "fix" 커밋
   - → 경고

## 모델 영향

- 추측이 반복되는 영역은 **모델 격상** 고려 (sonnet → opus)
- commit_perf_optimization의 모델 스위치에서 "추측 hit이 있던 영역은 격상"
  로직 검토

## 위반 시 행동

1. 즉시 작업 중단
2. 첫 행동 3원칙 중 하나로 돌아가기 (관찰부터)
3. 추측이었음을 사용자에게 명시
4. 잘못된 수정이 적용됐으면 되돌리기 검토
