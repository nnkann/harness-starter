# No-Speculation 규칙

defends: P1

추측으로 수정을 시작하지 마라.

## 첫 행동 3원칙

문제에 대한 첫 행동은 다음 중 하나여야 한다:

1. **관찰** — 실제 상태를 직접 읽는다 (파일, 로그, git status, 실행 결과)
2. **재현** — 문제가 발생하는 최소 케이스를 만든다
3. **선행 사례 확인** — 내부 자료(`internal-first.md`)에서 유사 사례

## 금지

- "아마 X가 원인일 것"으로 시작해서 X를 바로 고치는 것
- 원문(에러 메시지·로그·소스)을 보기 전에 해결책을 설계하는 것
- 테스트·실행 없이 "이제 될 것"이라 결론내리는 것
- 사용자에게 "이걸로 될까요?"라고 검증 책임을 떠넘기는 것

## 확신 표현

확신이 없으면 명시적으로:
- "가설이다, 확인 필요"
- "X일 가능성이 있는데, 로그를 봐야 확정"
- "지금까지 본 것만으로는 단정 불가"

## 에러·디버그 에스컬레이션

에러·예상 밖 동작 발생 시:

| 상황 | 행동 |
|------|------|
| 에러 원인이 1회 관찰로 불명확 | debug-specialist 즉시 |
| 동일 수정 2회 이상 반복 | debug-specialist 즉시 |
| 동일 시스템 동작 이슈 2회 이상 | debug-specialist 즉시 |
| 원인 알지만 해결 방향 불명 | debug-specialist 즉시 |

debug-specialist 호출 후에도 미해결이면 사용자 보고. 추측 재시도 금지.
