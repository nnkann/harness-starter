# Self-Verify 규칙

에이전트는 커밋 전이 아니라, **작업 중에** 자기 결과물을 검증해야 한다.
Review Agent(커밋 시점)에서 처음 문제를 발견하면 늦다.

## 원칙

"에이전트에게 검증 수단을 주면, 대부분 스스로 실수를 고친다." — Hashimoto

## 검증 시점

- 파일을 새로 만들거나 수정한 직후
- 함수/모듈 단위 작업이 끝날 때마다
- 사용자에게 "완료" 보고하기 전

## 검증 항목

**필수 (매번):** 빌드/컴파일 무결, 린터 0, 기존 테스트 통과.

**가능하면:** dev 서버 부팅, 변경 기능 실제 동작, 영향 범위 회귀 테스트.

## test-strategist 연계

self-verify는 **언제 검증해야 하는지**의 원칙. 어떤 테스트가 필요한지의
판단·추천은 **test-strategist 에이전트** 담당.

### 호출 트리거

- 새 함수·모듈·기능 추가에 테스트 없거나 부실
- 버그 수정 커밋에 회귀 방지 테스트 없음
- 리팩토링 전 안전망 확보 검토
- 기존 테스트가 flaky·실패 반복

### SKIP

- 단순 문서·설정·타이포 변경
- 기존 테스트 충분 + 변경 사소
- 테스트 코드 자체의 스타일 리뷰 (review 영역)

### 자동 호출

commit 스킬이 pre-check `needs_test_strategist` 신호로 자동 호출.
self-verify는 트리거 hit 명시만 하면 호출자(스킬·메인)가 실제 호출 수행.

## pipeline-design 체크리스트 연계

다단 처리 파이프라인 단계를 설계·재편할 때는 `.claude/rules/pipeline-design.md`
의 **7항목 체크리스트**를 작업 완료 전 반드시 검토:

1. 입력 / 2. 중간 계산 / 3. 출력 / 4. 폐기 / 5. 보존 책임 / 6. 전제 /
7. 검증 케이스

누락된 항목이 있으면 **"완료" 보고 금지**. 단계간 신호 흐름 설계는
단일 단계 내부 검증만으로는 잡히지 않는 실수가 흔함 (incident
`hn_pipeline_design_rule_origin.md`).

### 트리거

- 새 파이프라인 단계 설계 문서 작성
- 기존 단계의 입력·출력 구조 변경
- 상류 출력을 하류에서 재사용하는 구조 도입·삭제

### SKIP

- 단일 함수 내부 리팩토링 (단계 간 인터페이스 무변경)
- 파이프라인 외부 코드 수정

## 행동 규칙

- 검증 없이 "완료"라고 말하지 마라
- 실패하면 스스로 고치고 다시 검증
- 3회 시도해도 안 되면 사용자 보고. 추측으로 넘어가지 마라
- 검증 결과를 숨기지 마라. 실패했으면 실패했다고
- test-strategist 트리거 hit인데 호출 안 하고 "완료" 보고는 위반
- pipeline-design 7항목 누락 상태로 "완료" 보고는 위반
