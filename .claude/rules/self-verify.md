# Self-Verify 규칙

## 검증 워크플로우

```
WIP task AC 전부 [x]
  └─ work-verify (이 규칙): 내가 만든 것이 동작하는가? (린터 + AC가 요구한 테스트만)
      └─ 통과하면 /commit 스킬 호출
           └─ commit-check (pre_commit_check.py): 커밋 가능한 상태인가? (린터·dead link·WIP 잔여)
               └─ review: 변경이 안전한가? (회귀·계약·스코프)
```

**AC 전부 [x] = self-verify 통과 기준.** 체크박스가 남아 있으면 미완료다.

각 단계는 역할이 다르다. work-verify를 통과한 테스트는 같은 세션에서
코드 변경 없으면 재실행하지 마라 — commit-check이 대신하지 않는다.

에이전트는 커밋 전이 아니라, **작업 중에** 자기 결과물을 검증해야 한다.
Review Agent(커밋 시점)에서 처음 문제를 발견하면 늦다.

**다단계 작업 계획 형식**:
```
1. [파일 A 수정] → verify: AC 항목 1 실행
2. [파일 B 수정] → verify: AC 영향 범위에 명시된 marker 실행
3. [문서 갱신]   → verify: AC 항목 2 확인
```

## 원칙

"에이전트에게 검증 수단을 주면, 대부분 스스로 실수를 고친다." — Hashimoto

**Goal-Driven Execution**: 성공 기준(AC)을 먼저 정의하고 구현하라.

- WIP task AC 체크박스가 이 작업의 완료 기준이다
- 구현 후 AC 항목을 직접 실행·체크한다. "아마 됐겠지" 금지
- AC 없는 작업은 완료 선언 불가 — AC를 먼저 작성하라

## 검증 시점

- 파일을 새로 만들거나 수정한 직후
- 함수/모듈 단위 작업이 끝날 때마다
- 사용자에게 "완료" 보고하기 전

## 검증 항목

**필수 (매번):** 빌드/컴파일 무결, 린터 0.

**가능하면:** dev 서버 부팅, 변경 기능 실제 동작.

## 테스트 판단

> **무조건 매 커밋 pytest는 없다.** AC가 명시 요구할 때만 실행한다.

self-verify는 **언제 검증해야 하는지**의 원칙. AC가 트리거 SSOT.

### 트리거 매트릭스

| 상황 | 행동 |
|------|------|
| AC `영향 범위:` 항목에 `pytest -m <marker>` 표현 | 그 marker만 실행 |
| AC `영향 범위:` 항목에 `pytest <경로>` 표현 | 그 경로 실행 |
| AC가 테스트 요구 안 함 | 실행 안 함 (작성 의도가 그것) |
| 사용자 명시 요청 (`/test`, "테스트 돌려줘") | 요청 범위 실행 |
| 신규 함수·기능 추가 | TDD/fail-first — 테스트 먼저 작성 |
| 버그 수정 | 먼저 실패하는 회귀 테스트 작성, 그 다음 수정 |

### marker (test_pre_commit.py)

- `secret` (3): 시크릿 스캔 회귀 가드
- `gate` (2): completed 전환 차단 룰
- `stage` (8): pre_commit_check.py stage 결정 + move commit
- `enoent` (12): 린터 ENOENT 정규식 회귀 가드
- `docs_ops` (15): dead link / relates-to / wip-sync

전체 실행은 CI · `eval --deep` · 사용자 명시 요청 한정.

### TDD / fail-first

테스트를 먼저 작성하고 실패를 확인한 뒤 구현한다. AC가 테스트의 성공
기준이다 — AC 없이 테스트부터 짜지 마라.

### SKIP

- 단순 문서·설정·타이포 변경
- AC가 테스트 요구 안 한 변경 (자동 회귀 검증 대상 아님)
- 테스트 코드 자체의 스타일 리뷰 (review 영역)

## pipeline-design 체크리스트 연계

다단 처리 파이프라인 단계를 설계·재편할 때는 `.claude/rules/pipeline-design.md`
의 **7항목 체크리스트**를 작업 완료 전 반드시 검토:

1. 입력 / 2. 중간 계산 / 3. 출력 / 4. 폐기 / 5. 보존 책임 / 6. 전제 /
7. 검증 케이스

누락된 항목이 있으면 **"완료" 보고 금지**. 단계간 신호 흐름 설계는
단일 단계 내부 검증만으로는 잡히지 않는 실수가 흔함 (incident
`hn_pipeline_design_rule_origin.md`).

### 트리거

- 새 파이프라인 단계 설계 문서 작성
- 기존 단계의 입력·출력 구조 변경
- 상류 출력을 하류에서 재사용하는 구조 도입·삭제

### SKIP

- 단일 함수 내부 리팩토링 (단계 간 인터페이스 무변경)
- 파이프라인 외부 코드 수정

## 자동화 불가 검증 처리 원칙

검증 대상 중 **자동화할 수 없는 것**(Claude 행동·UI 동작·운용 시 효과)이
포함된 경우:

1. "자동 검증 불가 — 운용에서 확인 필요"를 명시적으로 고지한다
2. 테스트가 커버하는 범위와 커버하지 않는 범위를 구분해서 사용자에게 전달한다
3. **자동화 불가 검증을 자동화한 것처럼 포장하지 마라** — 통과한 테스트가
   실제로 무엇을 검증하는지 확인하지 않고 "검증됐습니다"로 단락하는 것이 위반

예시:
- ❌ "테스트 54/54 통과 → 검증됐습니다" (테스트가 규칙 텍스트 효과를 커버 못 함)
- ✅ "테스트 54/54 통과 — pre-commit 로직 검증 완료. Claude 행동 변화는 운용에서 확인 필요"
- ✅ "SKILL.md Step 절차 수정 — 스크립트 로직 테스트 통과. Claude가 실제로 수정된 절차를 따르는지는 운용에서 확인 필요"

## 행동 규칙

- 검증 없이 "완료"라고 말하지 마라
- 실패하면 스스로 고치고 다시 검증
- 3회 시도해도 안 되면 사용자 보고. 추측으로 넘어가지 마라
- 검증 결과를 숨기지 마라. 실패했으면 실패했다고
- pipeline-design 7항목 누락 상태로 "완료" 보고는 위반
