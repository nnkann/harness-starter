# Self-Verify 규칙

에이전트는 커밋 전이 아니라, **작업 중에** 자기 결과물을 검증해야 한다.
Review Agent(커밋 시점)에서 처음 문제를 발견하면 늦다.

## 원칙

"에이전트에게 검증 수단을 주면, 대부분 스스로 실수를 고친다." — Hashimoto

## 검증 시점

- 파일을 새로 만들거나 수정한 직후
- 함수/모듈 단위 작업이 끝날 때마다
- 사용자에게 "완료"라고 보고하기 전

## 검증 항목

### 필수 (매번)
- 빌드/컴파일이 깨지지 않는가?
- 린터 에러가 0인가?
- 기존 테스트가 통과하는가?

### 가능하면 (환경이 허락할 때)
- 앱이 정상 실행되는가? (dev 서버 부팅 확인)
- 변경한 기능이 실제로 동작하는가? (수동 확인 또는 테스트)
- 관련 없는 영역이 깨지지 않았는가? (영향 범위 테스트)

## 테스트 전략 — test-strategist 에이전트 연계

self-verify는 **언제 검증해야 하는지**의 원칙. 어떤 테스트가 필요한지의
판단·추천은 **test-strategist 에이전트**가 담당한다.

### 호출 트리거 (다음 중 하나라도 해당 시)

- **새 함수·모듈·기능 추가**에 테스트가 없거나 부실
- **버그 수정** 커밋에 회귀 방지 테스트가 없음 (같은 버그 재발 위험)
- **리팩토링 전** 안전망 확보 검토 (기존 테스트가 충분한지)
- 기존 테스트가 **flaky·실패 반복**해서 전략 재검토 필요

### 호출 흐름

```
1. self-verify 필수 항목 통과 확인
2. 위 트리거 hit 시 test-strategist 호출
   prompt에 다음 포함:
   - 분석 대상 (파일 경로 또는 변경 영역)
   - 목적 ("테스트 누락 식별" 등)
   - 맥락 (어떤 변경·버그·기능)
3. test-strategist의 권장 테스트 케이스 받음
4. 우선순위 높음 케이스부터 self-verify가 실제로 작성·실행
5. 사용자에게 완료 보고 시 "test-strategist 분석 + 추가 테스트 N개" 명시
```

### SKIP

- 단순 문서·설정·타이포 변경 (test-strategist 부르지 마라)
- 기존 테스트가 이미 충분하고 변경이 사소
- 테스트 코드 자체의 스타일 리뷰 (review의 영역)

### test-strategist는 단독 에이전트지만 self-verify가 자동 트리거

test-strategist를 사람이 일부러 부르기는 어렵다. **self-verify가 위
트리거를 보고 자동 호출**하는 게 정상 흐름. self-verify가 "에이전트
호출 권유"를 명시하면 호출자(스킬·메인 에이전트)가 실제 호출을 수행.

## 행동 규칙

- 검증 없이 "완료"라고 말하지 마라.
- 검증에 실패하면 스스로 고치고 다시 검증하라.
- 3회 시도해도 해결 안 되면 사용자에게 보고하라. 추측으로 넘어가지 마라.
- 검증 결과를 숨기지 마라. 실패했으면 실패했다고 말하라.
- test-strategist 트리거 hit인데 호출 안 하고 "완료" 보고는 위반.
