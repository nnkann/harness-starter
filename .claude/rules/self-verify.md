# Self-Verify 규칙

defends: P6

## 검증 워크플로우

```
WIP task AC 전부 [x]
  └─ work-verify (이 규칙): 내가 만든 것이 동작하는가? (린터 + AC가 요구한 테스트)
      └─ /commit 스킬 호출
           └─ commit-check (pre_commit_check.py): 커밋 가능한 상태인가?
               └─ review: 변경이 안전한가? (옵트인)
```

**AC 전부 [x] = self-verify 통과 기준.** 체크박스가 남아 있으면 미완료다.

## 원칙

성공 기준(AC)을 먼저 정의하고 구현하라. AC 없는 작업은 완료 선언 불가.

- WIP task AC 체크박스가 완료 기준
- 구현 후 AC 항목을 직접 실행·체크. "아마 됐겠지" 금지
- UI/frontend 변경 시: dev 서버 부팅 + 실제 동작 확인
- 현재 wave 안에서 확인할 수 없는 조건(예: 30일 후, 외부 파이프라인 가동 후,
  별 wave에서 확인)은 AC 완료 기준이 아니다. `blocked`·후속 WIP·운용 검증
  후보로 분리하고 완료로 포장하지 않는다.

## 검증 시점

- 파일 새로 만들거나 수정한 직후
- 함수/모듈 단위 작업 끝날 때
- 사용자에게 "완료" 보고 전

## 테스트 판단 — AC가 트리거 SSOT

| 상황 | 행동 |
|------|------|
| AC `검증.tests: pytest -m <marker>` | 그 marker만 실행 |
| AC `검증.tests: pytest <경로>` | 그 경로 실행 |
| AC `검증.tests: 없음` | 실행 안 함. 단, 왜 회귀 가드가 불필요한지 `실측`으로 닫아야 함 |
| 사용자 명시 요청 | 요청 범위 실행 |
| 신규 함수·기능 | TDD/fail-first — 테스트 먼저 |
| 버그 수정 | 회귀 테스트 먼저, 수정 그 다음 |

memory/reminder가 과거 회귀를 띄우면 테스트 선택의 **후보 신호**로만 쓴다.
현재 코드·문서·git log와 대조해 여전히 맞는 경우에만 AC `tests` 또는 `실측`에
반영한다. stale signal·count·오래된 PASS/FAIL 기록은 테스트 범위의 단독 근거가
아니다.

회귀 신호 판단:

- 현재 C와 같은 증상·같은 entry point면 P6/S6 기준으로 재현 또는 회귀 테스트를
  먼저 닫는다.
- 과거 incident가 있는데 작업 시점에 환기되지 않았으면 P8/S8 후보로 기록한다.
- 환기된 signal이 현재 코드와 다르거나 라벨·count·PASS·자가 선언이 baseline을
  오염시키면 P9/S9 후보로 보고 stale 처리한다. P9가 primary면 회귀 가드를
  기본값으로 두고, 문서·출력 계약처럼 테스트가 맞지 않는 경우에는
  재오염 방지 실측을 AC에 남긴다.
- 같은 구조가 여러 위치에 잠복하면 P11/S11 후보 위치를 함께 확인한다.

### marker (test_pre_commit.py)

- `secret` — 시크릿 스캔
- `gate` — completed 봉인
- `stage` — pre_commit_check.py
- `enoent` — 린터 ENOENT 회귀
- `docs_ops` — dead link / wip-sync

전체 실행은 CI · 사용자 명시 요청 한정.

## 자동화 불가 검증 처리

Claude 행동·UI 동작·운용 시 효과는 자동화 불가:

1. "자동 검증 불가 — 운용에서 확인 필요" 명시
2. 테스트 커버 범위·미커버 범위 구분 전달
3. **자동화 불가 검증을 자동화한 것처럼 포장 금지**
4. `tests: 없음`·`실측: 없음`·`운용 검증`은 완료 증거가 아니라 미커버 범위
   선언이다. 현재 wave의 문서·코드·출력 변화로 확인 가능한 최소 실측을 먼저 찾는다.

예시:
- ❌ "테스트 54/54 통과 → 검증됐습니다"
- ✅ "테스트 54/54 통과 — pre-commit 로직 검증 완료. Claude 행동 변화는 운용에서 확인 필요"
- ❌ "tests: 없음 / 실측: 없음 → 문서 작업이라 완료"
- ✅ "tests: 없음 — 문서 계약 변경. 실측: kickoff S6 해결 기준에 문구 반영, WIP AC가 S6을 인용"

## 행동 규칙

- 검증 없이 "완료" 금지
- 실패하면 스스로 고치고 재검증
- 3회 시도해도 안 되면 사용자 보고. 추측 금지
- 검증 결과를 숨기지 마라
- PASS/WARN/SKIP 라벨을 단독 증거로 쓰지 마라. 무엇을 검증했는지와
  무엇을 검증하지 못했는지를 함께 보고하라.
