# Pipeline Design 규칙

defends: P6

다단 처리 파이프라인 설계·재편 시 **상류 신호 재사용**과 **하류 보존
책임**을 체크리스트로 강제한다. 배경·업스트림 사료는 `docs/incidents/
hn_pipeline_design_rule_origin.md` 참조.

## 정의 — 본 규칙이 적용되는 "파이프라인"

다단 처리 구조 — 여러 단계가 **입력 → 중간 신호 → 출력**으로 연결되고,
각 단계가 다음 단계가 재사용 가능한 신호를 계산하는 구조.

**예시**:
- 데이터 변환 (ETL: extract → transform → load)
- ML/신호 처리 (전처리 → 특징 추출 → 분류 → 후처리)
- 에이전트 체인 (분석 → 판단 → 실행)
- 빌드 (컴파일 → 링크 → 패키징)
- 사용자 정의 단계 (프로젝트별 네이밍)

단순 함수 호출 체인(같은 축 판단을 반복하지 않음)은 대상 아님.

## 배경 — 왜 이 규칙이 필요한가

다단 처리 파이프라인에서 자주 발생하는 설계 실수: 어떤 단계가 풍부한
중간 신호(여러 축을 통합한 스코어 등)를 계산해 **한 번의 결정**에만
사용하고 **출력 구조에서 폐기**. 뒷 단계가 같은 축의 판단을 다시 해야
할 때 **열등한 정보로 재계산**하게 됨.

이 실수는 여러 차례 재설계를 거쳐도 발견하기 어렵다. 각 재편이 독립적
으로 합리적으로 보이지만, 공통 전제("상류에서 계산된 풍부한 신호는
이 단계에 필요 없다")가 틀렸기 때문. 한 달 이상 재편하면서도 이 패턴
을 의심하지 않은 실측 사례가 여러 프로젝트에서 관찰됨.

실제 발생 사례는 각 프로젝트의 `docs/incidents/*_pipeline_origin*`
참조. 업스트림이 이 규칙을 정립한 최초 사례: `docs/incidents/
hn_pipeline_design_rule_origin.md`.

## 원칙 — 단계 설계의 양방향 책임

### 받는 쪽 (upstream 활용)

1. **상류가 계산한 신호가 뭔지 먼저 파악** — 없다고 가정하고 시작하면
   중복 계산 발생
2. **같은 축 판단을 재계산하지 마라** — 상류 신호를 재활용하면 하류는
   결정만 담당
3. **전제 검증** — "이 신호가 이 단계에 필요 없다"는 판단의 근거를
   명시. 없으면 재활용 시도

### 넘기는 쪽 (downstream 보존)

1. **하류가 쓸 수 있는 중간 신호를 폐기하지 마라** — 결정에만 쓰고
   버리면 하류는 같은 축을 재계산
2. **출력 구조에 원본 축을 보존** — 단일 스칼라만 넘기지 말고, 결정의
   근거가 된 축 값을 같이 넘김
3. **보존 책임 명시** — 어떤 신호를 어디까지 전달하는지 인터페이스
   문서에 명기

## 금지 패턴

1. **좋은 도구 한 번 쓰고 버림** — 상류가 계산한 풍부한 신호를 한
   번의 분기 판단에만 쓰고 출력에서 제거
2. **상류 출력 암묵적 폐기** — 상류가 넘긴 신호를 하류가 "필요 없다"
   고 단정하고 전제 검증 없이 버림
3. **전제 미검증 재편** — "이 단계에 이 신호가 필요 없다"는 전제를
   여러 차례 재편에도 의심하지 않음
4. **단일 케이스 over-fit** — 특정 입력 케이스 하나를 기준으로 단계
   간 신호 흐름을 설계. 다른 케이스에서 발생할 재계산 비용 무시

## 설계 리뷰 체크리스트 (7항목)

단계 설계 문서 또는 재편 계획 작성 시 다음 7항목을 명시 의무. 빈 항목
있으면 self-verify.md가 "완료 보고 금지"를 적용.

1. **입력** — 이 단계가 받는 신호 목록 (상류 출력 그대로)
2. **중간 계산** — 이 단계 내부에서 새로 계산하는 신호 (판단·분류·
   스코어링 등)
3. **출력** — 하류로 넘기는 신호 목록
4. **폐기** — 이 단계에서 **의도적으로 버리는** 입력·중간 신호 +
   **왜 버리는지** 근거
5. **보존 책임** — 하류가 재계산할 가능성이 있는 축을 이 단계가
   보존하는지 확인
6. **전제** — "이 단계가 X 없이 작동한다"의 X 목록. 각 전제가 어느
   재편에서도 유효한지 검증
7. **검증 케이스** — 여러 입력 케이스에서 같은 설계가 성립하는지.
   단일 케이스 over-fit 방지

## 위반 감지

- **rule 체크리스트 강제** — 단계 설계 문서(예: `*_pipeline_*.md`·
  `*_stage_*.md`)에 위 7항목 명시 의무. 누락 시 self-verify.md가
  "완료 보고 금지" 적용
- **사용자 질문 트리거** — 같은 판단이 여러 단계에서 반복되는 패턴
  발견 시 즉시 "상류에 이 판단 신호가 있지 않나?" 질문
- **review 자동 감지는 본 규칙에 도입하지 않음** — 의도 설계 문제는
  diff 키워드 매칭으로 잡기 어려워 오탐 다수. 인지 도구(rule) + 사람
  질문 품질이 더 효과적 (`hn_review_staging_rebalance.md`·
  `hn_review_tool_budget.md` 결정과 정합)
