# 코딩 컨벤션

프로젝트의 코드 스타일과 패턴을 정의한다.
naming-convention이 "이름"을 다루면, 이 스킬은 "코드 자체"를 다룬다.

## 전제

- harness-init에서 스택이 이미 결정되어 있어야 한다.
- 린터/포매터가 선택되어 있어야 한다.
- 이 스킬의 결과물은 두 곳에 반영된다:
  - 린터로 강제할 수 있는 것 → 린터 설정 파일
  - 린터로 못 잡는 것 → .claude/rules/coding.md

## Standards 행동 원칙 (이 스킬 전체에서 적용)

스택 결정 이후 코딩 패턴을 정할 때, 업계 표준을 참조한다.

- 근거를 대라. "요즘 다 이렇게 해요"는 근거가 아니다.
- 맥락을 고려하라. 대기업 패턴을 소규모에 강요하지 마라.
- 피해야 할 안티패턴과 그 이유를 제시하라.
- 직접 구현 vs 라이브러리: 유지보수, 커뮤니티, 번들 크기, 대안을 함께.
- 확실하지 않으면 모른다고 하라.

## 설정 흐름

### 1. 기존 설정 확인

프로젝트에 이미 린터 설정이 있으면 먼저 읽는다.
있는 규칙을 중복으로 만들지 않는다.

### 2. 범용 기본 규칙 (언어 무관, 기본 적용)

아래는 사용자 확인 없이 기본 적용하는 규칙이다.
사용자가 변경을 원하면 그때 조정한다.

#### 변수/상수 케이스
- 상수: UPPER_SNAKE_CASE (`MAX_RETRY_COUNT`, `API_BASE_URL`)
- 변수: 소문자 시작 camelCase 또는 snake_case (스택 관례 따름)
- private/내부용: `_` 접두사 (`_cachedResult`, `_validateInput()`)
- 불변 참조(const): 일반 변수와 동일. 상수와 구분. const가 곧 상수는 아니다.

#### 함수 작성
- 한 함수는 한 가지 일만 한다 (Single Responsibility).
- early return으로 중첩을 줄여라. 깊은 if-else 금지.
- guard clause를 최상단에. 정상 흐름은 함수 끝에.
- 매직 넘버 금지. 상수로 추출하라.
- 순수 함수를 선호하라. 부작용이 있으면 함수명에 드러내라.

#### 코드 탐색
- grep 대신 LSP를 우선 사용하라 (정의 이동, 참조 찾기, 심볼 검색).
- LSP로 안 되는 경우에만 grep/ripgrep.
- 파일 전체를 읽지 말고 export 시그니처부터.

#### 일반 원칙
- 주석 처리된 코드 방치 금지. 삭제하라. git이 기억한다.
- console.log/print 방치 금지. 디버깅 끝나면 제거.
- TODO/FIXME/HACK 주석 금지. 미완료 작업은 코드가 아니라 docs/WIP/에 기록하라.
- 같은 로직 3번 반복되면 추출하라. 2번은 허용.

#### 프로세스/리소스 정리
- 테스트 코드에 종료 구문 필수. 프로세스가 좀비로 남으면 안 된다.
- 서버, DB 연결, 파일 핸들 등 열었으면 반드시 닫아라.
- 테스트 후 임시 파일/데이터 정리 (teardown/cleanup).

### 3. 카테고리별 규칙 결정

각 카테고리에서 선택지를 제시하고, 사용자가 결정한다.
스택에 따라 해당되지 않는 카테고리는 건너뛴다.

#### 임포트/모듈
- 임포트 정렬 순서 (예: 외부 → 내부 → 상대경로)
- 배럴 파일(index) 사용 여부
- 임포트 경로 제한 (예: 도메인 간 직접 임포트 금지)
- default export vs named export

#### 에러 처리
- 에러 처리 패턴 (try-catch 범위, 커스텀 에러 클래스)
- 에러 전파 규칙 (어디서 잡고 어디서 던지는가)
- 로깅 규칙

#### 타입 (타입 시스템이 있는 경우)
- any 사용 금지 여부
- 타입 추론 vs 명시적 타입 선언 기준
- 유틸리티 타입 사용 범위

#### 상태 관리 (프론트엔드인 경우)
- 전역 vs 로컬 상태 기준
- 서버 상태 관리 패턴

#### 금지 패턴
- 스택 공통: console.log 방치, 매직 넘버, 주석 처리된 코드 방치
- 추가 금지 패턴은 사용자에게 확인

### 4. 강제력 설계

결정된 규칙을 방어 레이어에 배치한다.
**이것이 이 스킬에서 가장 중요한 단계다.**

| 분류 기준 | 배치 | 예시 |
|----------|------|------|
| 린터 규칙으로 자동 감지 가능 | 린터 설정 파일 | no-console, no-any, import-order |
| 린터로 못 잡지만 반복 위반 위험 | .claude/rules/coding.md | 에러 전파 규칙, 추상화 기준 |
| 가이드 수준 (강제 불필요) | .claude/rules/coding.md (참고 섹션) | 선호하는 패턴 |

**원칙: 린터로 잡을 수 있는 건 반드시 린터에 넣는다.**
rules/에만 적어두면 무시된다. CLAUDE.md에만 적어두면 더 무시된다.

강제력 배치 점검표:
- [ ] "~하지 마라" 형태의 규칙 → 린터 no-* 규칙으로 변환 가능한가?
- [ ] 임포트/정렬 규칙 → 린터 플러그인으로 강제 가능한가?
- [ ] 타입 관련 규칙 → tsconfig strict 옵션으로 잡히는가?
- [ ] 파일명 패턴 → 린터 커스텀 규칙 또는 pre-commit hook으로 잡히는가?

### 5. 반영

- 린터 설정 파일 업데이트
- .claude/rules/coding.md 생성 또는 업데이트
- 사용자에게 요약 보여주고 확인

## .claude/rules/coding.md 구조

```markdown
# 코딩 컨벤션 (린터가 못 잡는 것)

## 패턴
(린터로 강제할 수 없지만 따라야 할 패턴)

## 금지
(린터로 감지할 수 없는 금지 패턴)

## 참고
(강제는 아니지만 권장하는 가이드)
```

## 주의

- 이 스킬은 범용이다. 카테고리는 고정이지만 선택지는 스택에 따라 다르다.
- 모든 규칙을 한 번에 정할 필요 없다. 중요한 것부터 정하고 나머지는 개발하면서 추가.
- 승격 규칙 적용: 위반 반복 시 rules → 린터로 승격.
