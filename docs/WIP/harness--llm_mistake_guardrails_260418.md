# LLM 실수 방지 가드레일

## 배경

최근 하네스 개발 중 동일한 실패 패턴이 반복 관찰됨:

1. **허위 진단 → 즉시 행동**: 확신 없는 가설을 "확정"으로 쓰고 바로 우회/수정
2. **성급한 단정**: 1~2회 테스트로 "X는 불가능"이라 결론, 변형 조건 미검증
3. **선행 성공 사례 무시**: git history에 작동 사례가 있는데 새로 설계
4. **같은 영역 연속 수정**: 증상 완화만 반복, 근본 원인 미도달
5. **허위 후속 작업 생성**: "재검토 필요" 같은 모호한 후속을 문서에 박음

이 문서는 코드/규칙으로 강제 가능한 방지 장치를 정리한다.

## 제안하는 방지 대책

### 1. "확정" 단어 사용 조건 (CLAUDE.md 규칙)

`.claude/rules/`에 추가:

> **"X가 불가능하다/안 된다"로 결론내기 전 반드시:**
> - 공식 문서 또는 선행 구현 성공 사례를 근거로 제시
> - 최소 2가지 변형 조건으로 재현 (예: `if` 유무, matcher 단순화)
> - 실패 케이스를 증명할 수 있는 테스트 설계
>
> 위 3개 중 하나라도 못 했으면 "확정" 대신 "현재 조건에서 관찰됨"으로 표기.
> WIP·커밋 메시지·promotion-log의 "확정"·"증명"·"불가능" 단어는 위 근거와 함께만 사용.

### 2. 선행 성공 사례 우선 탐색 (CLAUDE.md 규칙)

> 새 구조를 제안하기 전, 반드시:
> - `git log --all --oneline | grep <관련 키워드>` 로 과거 관련 커밋 확인
> - 사용자가 "예전엔 됐다"는 증언이 있으면 그 시점 찾기가 최우선
> - 해당 시점 파일 내용을 실제로 체크아웃/show 하여 확인

### 3. 허위 진단 자동 감지 (review agent 확장)

review.md에 검증 항목 추가:

> **허위 후속 작업 감지**: WIP 문서나 커밋 메시지에 다음 패턴이 있으면 차단:
> - "재검토 필요", "추후 확인", "검증 예정" 등 모호한 후속
> - 왜 재검토가 필요한지 구체적 근거가 없으면 [차단]
> - 근거가 있으면 "...이 이유로 N단계 후속 작업 필요"처럼 명시 요구

### 4. 같은 영역 연속 수정 감지 (pre-check 확장)

pre-commit-check.sh에 추가:

```bash
# 같은 파일이 최근 5개 커밋에 반복 등장하면 경고
RECENT=$(git log -5 --name-only --format= | sort | uniq -c | sort -rn)
REPEATED=$(echo "$RECENT" | awk '$1 >= 4 {print $2}')
if [ -n "$REPEATED" ]; then
  git diff --cached --name-only | while read f; do
    if echo "$REPEATED" | grep -qx "$f"; then
      echo "⚠️  $f는 최근 5커밋 중 4+회 수정됨. 근본 원인 미해결 가능성." >&2
    fi
  done
fi
```

차단이 아니라 경고만. 사용자 판단.

### 5. 즉시 우회 금지 (commit 스킬 규칙)

> `[skip-review]` 플래그 사용 조건:
> - 사용자가 명시적으로 지시한 경우만
> - 리뷰 에이전트가 차단했을 때 "템플릿 버그"로 추정하여 스킵 금지
> - 자동 스킵 절대 금지

## 구현 범위

| 항목 | 반영 위치 |
|------|----------|
| 1. "확정" 단어 조건 | `.claude/rules/` (신규 또는 기존 rule에 추가) |
| 2. 선행 사례 우선 | `.claude/rules/` |
| 3. 허위 진단 감지 | `.claude/agents/review.md` |
| 4. 연속 수정 감지 | `.claude/scripts/pre-commit-check.sh` |
| 5. 우회 금지 | `.claude/skills/commit/SKILL.md` |

## 우선순위

- **P0**: 1 + 4 — 가장 싸고 이번 실패 패턴에 직접적
- **P1**: 2 + 5 — 행동 규칙
- **P2**: 3 — review agent 확장은 검증 방식 설계 필요

## 주의

이 가드레일 자체도 과도하게 적용하면 정상 작업까지 느려진다.
특히 4번(연속 수정 감지)은 정당한 반복 작업(예: 같은 파일의 기능 확장을
단계별 커밋)에 오탐 가능. 경고 수준으로만 유지.
