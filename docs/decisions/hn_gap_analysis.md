# 하네스 엔지니어링 적합성 분석

영상(하네스 엔지니어링 개념 소개)의 핵심 요소와 현재 프로젝트(harness-starter v0.6.0)를 대조 분석한 결과.

## 영상의 3가지 핵심 vs 현재 상태

| 핵심 요소 | 영상 설명 | 현재 프로젝트 | 판정 |
|-----------|----------|-------------|------|
| 가시성 확보 | 로그/스크린샷으로 AI가 자기 결과물 확인 | session-start.sh, stop-guard.sh, post-compact-guard.sh | 충분 |
| 점진적 컨텍스트 | 목차 구성, 필요한 정보를 스스로 찾아가게 | CLAUDE.md -> rules/ -> skills/ -> docs/WIP/ 계층 구조 | 충분 |
| CI/CD 강제 | 규칙 위반 시 에러로 차단 | pre-commit-check.sh + auto-format.sh + write-guard.sh | 충분 |

## 이미 잘 갖춰진 부분

### 1. 4계층 강제력 배치

영상에서 말하는 "단순 지시를 넘어 에러로 강제"가 정확히 구현됨.

- Linter (물리 법칙) -> Hooks (법) -> Rules (자동 로드) -> CLAUDE.md (소원)
- `--no-verify` 차단까지 있어서 우회 경로도 막혀 있음

### 2. 자동 피드백 루프

영상의 핵심 주장 "인간 병목 제거"에 해당.

- 커밋 전 5단계 검증 (pre-commit-check.sh)
- 파일 저장 직후 포매팅 (auto-format.sh)
- 새 파일 생성 시 중복 확인 유도 (write-guard.sh)
- 세션 시작 시 상태 자동 표시
- 컨텍스트 압축 후 맥락 재주입 (post-compact-guard.sh)

### 3. 작업 문서 라이프사이클

영상에서는 언급 안 됐지만 이 프로젝트의 차별화 포인트.

- docs/WIP/ 기반 상태 추적 (pending -> in-progress -> completed)
- commit 스킬이 완료 문서를 자동 이동

### 4. 강도 분리 (light/strict)

프로젝트 규모에 맞는 유연성 제공.

## 영상 대비 모자란 부분

| 항목 | 영상 내용 | 현재 상태 | 개선 방향 |
|------|----------|----------|----------|
| 워크트리 분리 | 메인 코드를 건드리지 않고 별도 워크트리에서 테스트 후 병합 | CLAUDE.md에서 워크트리 금지 명시 | 아래 분석 참조 |
| 스크린샷/시각적 검증 | AI가 스크린샷을 찍어 결과물을 눈으로 확인 | 메커니즘 없음 | UI 프로젝트라면 Playwright 스크린샷 hook 추가 고려 |
| 테스트 자동 실행 | verify-tasks 스크립트로 커밋 전 테스트 필수 통과 | 린터만 강제, 테스트는 agent prompt로만 확인 | pre-commit-check.sh에 테스트 실행 단계 추가 가능 |
| 빌드 검증 | 빌드가 깨지지 않는지 커밋 전 확인 | self-verify.md에 규칙으로만 존재 (강제 아님) | 빌드 명령어 실행을 hook으로 승격 가능 |

### 워크트리 금지에 대한 분석

영상에서는 워크트리를 안전 장치로 강력 추천하지만, 이 프로젝트는 명시적으로 금지하고 있다.

- **영상의 논리**: 메인 브랜치를 오염시키지 않고 테스트 완료 후 병합 -> 안전
- **이 프로젝트의 논리**: 워크트리 관리 복잡성, 병합 충돌, Windows 환경 이슈 회피 -> 단순
- **현재 대안**: pre-commit-check.sh가 커밋 시점에서 차단하므로, 메인에서 직접 작업해도 "나쁜 커밋"은 방지됨

**결론**: 현재 접근법도 유효. 워크트리가 없는 대신 커밋 게이트가 충분히 강력함. 단, 대규모 리팩터링에서는 브랜치 분리를 수동으로 해야 하는 점은 인지 필요.

## 과한 부분

| 항목 | 현재 상태 | 과한 이유 | 권장 |
|------|----------|----------|------|
| 10개 스킬 | full 프로파일 기준 10개 | 코드가 없는 스타터 상태에서 전부 로드 | 프로파일 시스템이 있으므로 적절. eval, advisor는 코드 생긴 후 의미 있음 |
| 3관점 병렬 에이전트 (advisor) | 리서치+코드분석+리스크 3개 동시 호출 | 소규모 프로젝트에서는 토큰 낭비 | 온디맨드 구조이므로 현재 설계 적절 |
| strict 모드 Review Agent | 계획/CPS/품질/부작용 전부 검증 | 프로토타입이나 개인 프로젝트에서는 속도 저하 | light/strict 선택 가능하므로 과하지 않음 |
| 빈 플레이스홀더 rules | coding.md, naming.md가 비어있는 채로 매번 로드 | 컨텍스트 낭비 | harness-init 전에는 조건부 로드 검토 |

## 종합 판정

```
영상의 핵심 개념 커버율: 80%

잘 된 것:  강제력 계층, 자동 피드백 루프, 점진적 컨텍스트, 문서 라이프사이클
모자란 것: 시각적 검증(스크린샷), 테스트/빌드의 hook 강제
과한 것:  빈 플레이스홀더 파일 로드 (미미한 수준)
설계 차이: 워크트리 금지 (대안으로 커밋 게이트 강화 — 합리적 트레이드오프)
```

## 추천 액션 (우선순위순)

1. **테스트 실행을 hook으로 승격** — self-verify 규칙(소원)에서 pre-commit-check.sh(법)로. CLAUDE.md의 빌드/실행 명령어가 채워진 후 `npm test` 또는 동등 명령을 커밋 전 강제 실행
2. **UI 프로젝트 시 스크린샷 검증 추가** — Playwright가 이미 글로벌 설치되어 있으므로, PostToolUse hook에서 dev 서버 스크린샷을 캡처하는 스크립트 추가 가능
3. **빈 rules 파일 조건부 로드** — coding.md/naming.md가 비어있으면 로드 스킵하는 로직 (토큰 절약, 미미하지만 깔끔)

이 중 1번은 실제 프로젝트 코드가 생기면 바로 적용할 수 있고, 2/3번은 상황에 따라 선택적으로 적용.
