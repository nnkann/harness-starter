# 문서 탐색 프로토콜

`.claude/rules/docs.md`에서 분리. 매 세션 컨텍스트에 박을 필요 없는
**탐색 절차** 상세를 모은다. doc-finder 에이전트와 메인 Claude가 문서
검색 시 참조.

## 언제 탐색하는가

| 트리거 | 예시 |
|--------|------|
| "왜 이렇게 했지?" | "왜 Redis를 선택했어?", "이 설계의 근거가 뭐야?" |
| "어떻게 하지?" | "배포는 어떻게 해?", "업그레이드 절차가 뭐야?" |
| "전에 이런 적 있었나?" | "비슷한 버그 있었어?", "이거 해본 적 있어?" |
| 새 작업 시작 전 맥락 | implementation Step 0 |
| 결정 전 선례 확인 | 기술 선택, 아키텍처 변경 전 |

## 탐색하지 않는 경우

- 코드 수정만 하는 단순 작업 (타이포, 변수명)
- 사용자가 구체적 파일을 지정한 경우
- git log로 충분한 질문

## 탐색 절차

```
1. docs/INDEX.md → 도메인 식별
2. clusters/{domain}.md → 문서 목록 + 관계 맵
3. 관련 문서 본문 Read (보통 1~3개)
4. relates-to 포인터로 1홉 추가 (필요 시)
```

문서 ≤10개면 직접 탐색. 많거나 도메인 불확실하면 doc-finder 위임.

## IDE 컨텍스트는 힌트, 진실 아님

`<ide_opened_file>`, `<ide_selection>` 등은 실제 존재 여부 무관. 직전에
닫힌 파일·다른 워크스페이스·미저장 파일도 옴.

- IDE 경로를 검색 키로 쓰기 전 **Read 또는 Glob으로 존재 확인**
- 없으면 그 파일명 **버리고** 사용자 원문 키워드로 재검색
- IDE 파일명만으로 "없다" 결론 금지 — 사용자가 친 키워드가 아닐 수 있음

## "없습니다"는 3단계 이후에만

3단계를 모두 거쳐야 "없다"고 말할 수 있음:

```
1. 파일명 Glob        → Glob "**/*<keyword>*.md"
2. 제목/태그 grep     → Grep "title:.*<keyword>|tags:.*<keyword>" docs/
3. 본문 grep (필수)   → Grep -i "<keyword>" docs/ --include="*.md"
```

사용자 원문에서 **고유명사**(공연명, 아티스트, 업체, ID, 제품명)를 추출해
키워드로 사용. IDE가 준 파일명 단어가 아님.

## 검색 실패 escalation

3단계 모두 비었을 때:

```
3단계 빔
  → doc-finder 위임 (사용자 원문 + 추측 키워드)
    → 에이전트도 빔
      → 사용자에게 키워드/범위 재질의
```

"열심히 찾았는데 없다" 증명 책임은 Claude. 사용자에게 떠넘기기 전 위
단계 모두 밟았는지 확인.
