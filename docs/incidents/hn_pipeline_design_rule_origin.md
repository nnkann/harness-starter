# pipeline-design 규칙 업스트림 이식 원천 사례

## 발생 맥락

다운스트림 프로젝트(`<프로젝트 사례>`)가 detection pipeline 재편 중
다음 설계 실수를 한 달간 draft1 → draft2 → draft3로 재편하면서도
발견하지 못함:

> "상류 단계가 어텐션 스코어 등 **여러 축을 통합한 풍부한 신호**를
> 계산해 병합 결정에 사용하고 **출력 구조에서 폐기**. 하류 단계들이
> 같은 축(색 거리 등)의 판단을 다시 해야 할 때 **열등한 정보로 재계산**.
> 같은 판단이 3번 반복되는 구조."

각 재편이 독립적으로는 합리적이었지만, 공통 전제("하류는 상류가 계산한
풍부한 신호가 필요 없다")가 틀렸기 때문에 누적 비용이 계속 발생.

## 한 달간 재편하면서도 못 본 이유

LLM + 인간 조합의 공통 맹점:

1. **재편이 각 단계의 내부 개선에 집중** — 단계 **간** 신호 흐름을
   보는 관점이 자동으로 발동하지 않음
2. **상류 출력 구조는 "당연한 것"으로 간주** — 뭘 넘기는지 재검토하지
   않음
3. **LLM의 코드 변경 관점이 단계 내부에 편향** — diff 키워드로 전체
   파이프라인 신호 흐름을 추론하기 어려움
4. **단일 케이스 over-fit** — 테스트 케이스 하나로 "동작한다"를 검증
   하면 다른 케이스의 재계산 비용이 안 보임

## 업스트림 범용화 판단 근거

이 실수는 ML 파이프라인뿐 아니라 ETL·빌드·에이전트 체인 등 **다단
처리 전반에 적용 가능**:

- ETL: extract가 계산한 메타데이터를 transform에서 폐기 → load가 같은
  메타데이터 재계산
- 에이전트 체인: 분석 에이전트가 계산한 신뢰도 점수를 판단 에이전트가
  폐기 → 실행 에이전트가 다시 신뢰도 계산
- 빌드: 컴파일 단계의 의존성 그래프를 링크가 폐기 → 패키징이 재생성

범용 규칙으로 올려 미래 프로젝트의 동일 실수 방지.

## 업스트림 반영 결정 (2026-04-21)

- `.claude/rules/pipeline-design.md` 신규 — 범용 버전 (100줄,
  7항목 체크리스트)
- `CLAUDE.md`에 `<important if>` 트리거 추가 (다단 처리 파이프라인
  설계·재편 시)
- `.claude/rules/self-verify.md`에 "파이프라인 단계 설계·재편 시
  체크리스트 7항목" 항목 추가
- **review 자동 감지 패턴은 도입하지 않음** — 의도 설계 문제는 diff
  키워드 매칭으로 잡기 어려움. rule 체크리스트 강제 + self-verify
  연계가 더 효과적 (v0.17.0·0.17.1 review 재조정과 정합)

결정 문서: `docs/decisions/hn_pipeline_design_rule.md`

## 다운스트림이 받을 것

| 항목 | OK 사유 |
|---|---|
| 범용 rule 파일 | 원 프로젝트 100줄 일반화. 기능 유지 |
| CLAUDE.md `<important if>` | 조건 트리거, 파이프라인 작업 아니면 무시 |
| self-verify 연계 | 기존 흐름에 체크리스트 추가만 |

다운스트림 커스터마이징은 `pipeline-design.md` 하단 "프로젝트 고유
사례" 섹션(harness-upgrade 보존 영역)에 추가.

## 교훈

- **단계 **간** 신호 흐름**을 보는 관점은 자동 발동 안 함 → 체크리스트로
  강제 필요
- 재편 횟수로는 설계 실수가 덮이지 않음 — 공통 전제가 틀리면 재편이
  오히려 실수를 고착화
- 한 프로젝트의 hard-won lesson은 다른 프로젝트도 겪을 확률 높음 →
  업스트림 반영으로 미래 방지
